> **版本**：v1.0  
> **日期**：2026-06-30  
> **状态**：草稿  
> **负责人**：产品 / 研发

---

## 1. 背景与目标

### 1.1 背景
当前招聘流程中，简历内容容易注水，难以验证候选人的真实编码能力。现有系统已具备简历 AI 解析 + 智能匹配能力，但缺少对“实际编码能力”的验证环节。

### 1.2 目标
构建一套**面试者专用前端 + 受限评估后端**的闭环，实现：

- 面试者只能在专门设计的面试平台完成选题、计时、作品提交
- 面试官（受限角色）可在 HR 系统内对**代码作品**进行手动评估打分
- 面试者提交的简历自动流入候选人库
- **严格的角色隔离**：面试者绝对无法接触 HR 后台数据

**当前阶段只做“代码作品评估”**，不涉及完整面试流程、面试反馈、多轮面试等功能。

---

## 2. 用户角色定义（核心约束）

| 角色          | 中文名称   | 创建方式           | 登录后可见范围                          | 关键约束 |
|---------------|------------|--------------------|-----------------------------------------|----------|
| `admin`       | 内部管理员 | 仅后台创建         | 完整 HR 系统（看板、候选人、岗位、企业、匹配、AI 助手、账号管理等） | 拥有全部权限 |
| `interviewer` | 面试官     | **仅后台创建**（admin 操作） | 仅与候选人评估相关的受限界面（候选人列表 + 代码作品评估） | 不能创建用户、不能看到其他模块 |
| `interviewee` | 面试者     | **仅前端公开注册** | 仅面试平台界面（面试题列表、选题、计时、提交作品） | **绝对隔离**，禁止访问任何 HR 数据和接口 |

### 2.1 角色创建规则（硬性）
- 面试官账号**只能由 admin 在后台创建**，前端任何注册/自助流程都**禁止**创建 `interviewer` 或 `admin`。
- 所有通过前端注册流程创建的用户，**强制**角色为 `interviewee`。
- 系统必须在用户创建时和后续所有操作中强制校验角色。

## 3. 核心业务流程

### 3.1 面试者（Interviewee）流程

1. 进入面试平台公开页面
2. 注册账号（或已有账号登录）
3. 查看面试题列表（目前 4 道题：01~04）
4. 选择一道题
5. 系统记录：
   - 选择者（用户）
   - 所选题目
   - 开始计时（3 小时）
6. 在时限内上传：
   - GitHub 作品链接
   - 简历文件（PDF/Word 等）
7. 提交
8. 系统自动执行：
   - **简历自动入库** → 成为 `candidate` 记录，进入简历池
   - **作品提交记录** → 创建 `code_submission`，状态为 `pending_evaluation`，进入“待评估”池

**时限规则**（待确认细节）：
- 选择题目后开始 3 小时倒计时
- 超时后是否仍允许提交？（建议先允许提交但标记“超时”）

### 3.2 面试官（Interviewer）流程

1. 由 admin 在后台创建账号并分配 `interviewer` 角色
2. 使用该账号登录 HR 系统
3. 只能看到受限的“候选人评估”相关界面
4. 在“待评估”列表中看到：
   - 面试者基本信息
   - 所选题目
   - GitHub 链接
   - 上传的简历
5. 打开详情后手动填写：
   - 代码分数（0-100）
   - 可选等级（S/A/B/C）
   - 评语/备注
6. 保存后，记录评估人和评估时间，状态变为 `evaluated`

### 3.3 内部 HR（Admin）流程

- 使用完整权限账号登录
- 可查看所有候选人（包括从面试平台流入的）
- 可查看候选人关联的代码评估结果（分数、题目、链接等）
- 正常使用 HR 系统其他功能

## 4. 功能需求

### 4.1 面试者端（Interview Platform）

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F1 | 公开注册/登录 | 面试者可自助注册，注册后强制角色为 `interviewee` | P0 |
| F2 | 面试题展示 | 展示 4 道题目（01~04），包含题目描述、交付要求、加分项 | P0 |
| F3 | 选题与计时 | 选择一道题后记录选择，开始 3 小时倒计时，前端展示剩余时间 | P0 |

## 5. 数据模型建议（最小可行版）

### 5.1 新增表：`code_submission`（代码作品提交）

```sql
code_submission
- id (PK)
- user_id (FK user.id)                    -- 提交者（interviewee）
- challenge_id (varchar)                  -- "01" / "02" / "03" / "04"
- github_url (text)
- resume_attachment_id (FK attachment.id, nullable)
- candidate_id (FK candidate.id)          -- 简历入库后关联
- status (varchar)                        -- pending_evaluation / evaluated / timeout
- submitted_at (timestamptz)
- time_spent_seconds (int)                -- 从选题到提交的耗时（可选）
- score (numeric(5,2), nullable)          -- 0-100
- grade (varchar, nullable)               -- S / A / B / C
- evaluated_by (FK user.id, nullable)     -- interviewer
- evaluated_at (timestamptz, nullable)
- notes (text, nullable)
- created_at, updated_at
```

### 5.2 题目定义
初期可硬编码 4 道题（01~04），后续可考虑做成 `code_challenge` 表配置化。

### 5.3 对现有模型的影响
- `candidate` 表无需大改，可通过 `source` 或新增字段标记“来自面试平台”
- `user.role` 需新增支持 `interviewer`、`interviewee`（当前已有 `admin`/`consultant`）

## 6. 权限与安全要求（必须满足）

### 6.1 后端
- 所有接口使用 `get_current_user` + 角色校验
- 提供 `require_interviewer`、`require_interviewee` 等依赖
- `interviewee` 只能访问：
  - 自己的选题记录
  - 自己的提交记录
  - 题目列表
- 面试者**禁止**访问 `/candidates`、`/positions`、`/companies`、`/matches` 等任何路径

### 6.2 前端
- 根据 `role_name` 控制菜单显示
- 路由层增加角色守卫（未授权角色直接跳转或 403）
- 面试者端建议与主 HR 系统**物理或逻辑隔离**（可通过独立路由前缀 `/interview/*` + 严格拦截）

### 6.3 审计
- 关键操作记录日志（选题、提交、打分）

---

## 7. 非功能需求

- **安全性**：面试者绝对隔离是最高优先级需求
- **易用性**：面试者端流程简洁，3 小时计时清晰可见
- **可扩展性**：后续可增加自动代码评测（OpenClaw）、多题提交、超时策略等
- **数据一致性**：简历入库与作品提交需在同一事务或有补偿机制

## 8. 范围与排除范围

### In Scope（本期）
- 角色体系完善（admin / interviewer / interviewee）
- 面试者端选题 + 计时 + 提交

- 付费、会员、商业化功能
- 面试者端与 HR 系统的数据互通（除简历入库和代码分外）

## 10. 验收标准（简要）

1. admin 可创建 interviewer 账号
2. 前端注册的用户角色强制为 interviewee
3. interviewee 登录后只能看到面试题/提交界面，无法访问其他页面和接口
4. 面试者选题后 3 小时计时正常
5. 提交作品后，简历出现在候选人库，作品出现在待评估列表
6. interviewer 可对作品手动打分并保存
7. 候选人详情可看到对应的代码评估结果
