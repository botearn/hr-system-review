# 分析报告：架构与开发流程（试验期 v0）

> 本文目标：用“可升级的短周期产品”视角，客观记录当前系统在**功能承载、开发流程、架构可演进性**上的问题与风险。  
> 约束：**不要求现在改代码**，但给出明确的升级方向与优先级。

## 0. 执行摘要（给决策者）

当前版本能支撑 demo/试点：候选人/岗位/企业/跟进闭环 + 简历导入流水线 + Qdrant 匹配引擎 + 右侧 AI 助手。

但它也呈现出典型“短周期试验产品”的结构特征：  
- **流程层薄、数据口径与可观测性不足**（没埋点/没指标闭环）  
- **异步任务/LLM 调用缺少可靠性机制**（重试、幂等、队列、超时与隔离）  
- **领域边界尚未稳定**（HR/候选人语境混淆、模块入口变化快）  
- **工程护栏历史欠账较多**（CI/测试/代码规范之前未固化，已开始补齐）

建议策略：把系统分成“可演示的核心链路”与“可延后扩展的能力模块”，并在 2-4 周内补齐最低限度的**可靠性与可观测性**，让迭代成本不指数上升。

## 1. 功能与产品流程：当前问题（不评价对错，只评价风险）

### 1.1 激活路径与默认落点
- **现状**：存在独立的 `/agent` 聊天页，同时在非 `/agent` 页面右侧有 `AiPanel`。看板页已上线，但文档与入口策略经历过多次调整。
- **风险**：新用户第一次进入系统，可能无法立刻理解“系统的招牌价值是什么”（匹配/导入/跟进/统计），容易把它当成普通 admin。
- **升级方向**：
  - 固化“默认落点”与“一屏价值叙事”（看板或 agent 二选一）
  - 将核心动作做成明确 CTA：导入简历 → 岗位 → 匹配 → 跟进 → 复盘

### 1.2 角色与语境一致性（HR 侧 vs 候选人侧）
- **现状**：AI 助手偶尔会漂移到候选人第一人称，这会破坏产品可信度。
- **风险**：一旦出现“自称候选人/求职者”的回答，会显著降低 HR 用户对系统的信任。
- **升级方向**：
  - 在系统提示词、前端文案、推荐问题上持续强化“HR 助手语境”
  - 在关键 UI 处标记：系统建议 ≠ 候选人表达

### 1.3 指标口径（统计/漏斗）
- **现状**：看板/漏斗已存在，但“漏斗阶段定义”与数据来源（status_change vs candidate 字段）需要严格对齐。
- **风险**：如果口径漂移，系统会出现“看板数据与一线操作不一致”，后续很难纠正用户认知。
- **升级方向**：
  - 为每个指标写“口径定义”（来源表、过滤条件、时间范围）
  - 把口径作为 API 合同的一部分（文档 + 测试）

## 2. 架构分析：可演进性与主要风险点

### 2.1 单体后端 + 背景任务：适合试验，但可靠性欠账要可控
- **现状**：大量异步工作（简历解析、能力提炼、向量化、reindex）由 FastAPI BackgroundTasks 驱动。
- **优点**：实现快、部署简单，适合短周期验证。
- **风险**：
  - 进程重启/崩溃会丢任务
  - 无重试/无队列/无限并发可能拖垮服务
  - 幂等不足会导致重复写入或“半成功”状态
- **升级方向（从轻到重）**：
  1) 为任务引入明确状态机 + 重试策略（数据库可追踪）
  2) 引入队列（Redis + RQ/Celery/Arq 任一）并把任务迁移出去
  3) 对 LLM/Embedding/Qdrant 调用做隔离（超时、熔断、限流）

### 2.2 数据层：ORM 与原生 SQL 混用需要强约束
- **现状**：主要用 SQLAlchemy ORM，但某些统计使用 raw SQL。
- **风险**：表名/字段名不一致会造成线上故障（尤其迁移/重构时）。
- **升级方向**：
  - 统计尽量用 ORM 或封装后的 SQL；raw SQL 必须集中在一个模块并配测试
  - 对关键 SQL 写 “migration + query” 的契约测试

### 2.3 LLM/Embedding：不可控依赖需要可观测性
- **风险表现**：慢、超时、返回格式漂移、偶发失败、模型升级导致行为改变。
- **升级方向**：
  - 统一 LLM 客户端：超时、重试、日志、request_id、成本统计
  - “结构化输出”失败时必须可降级（比如回退到纯文本 + 手动确认）

### 2.4 前端：多页面 + 右侧 AI Panel 的一致性
- **现状**：`/agent` 与右侧 `AiPanel` 是两套体验。
- **风险**：两套对话历史与推荐问题策略不同，容易造成“用户不确定该用哪一个”。
- **升级方向**：
  - 明确定位：`/agent` 作为专业入口，`AiPanel` 作为上下文助手
  - 统一推荐问题策略与对话历史持久化策略（localStorage/sessionStorage/后端存储）

## 3. 开发流程分析：短周期可接受，但要避免“协作熵增”

### 3.1 代码规范与自动化护栏
- **现状**：已补齐 `.editorconfig`、前端 prettier/eslint/typecheck、后端 ruff format/check，并新增 CONTRIBUTING 与 Cursor rules。
- **风险**：若不在 CI/PR 中强制执行，规范会逐渐失效。
- **升级方向**：
  - 最低限度引入 CI：frontend `lint+typecheck`、backend `ruff+pytest`
  - 引入 pre-commit（可选）提升本地一致性

### 3.2 测试策略
- **现状**：有脚本级测试（seed_demo/test_matching），缺少 API 级回归与关键路径测试。
- **风险**：LLM 与异步任务让回归更难，越往后越不敢动。
- **升级方向（建议最小集合）**：
  - 单测：纯函数（打分、过滤、归一化）
  - 契约测试：关键 API 响应 shape
  - 冒烟测试：导入 → 确认 → 匹配 → 跟进 → 看板

### 3.3 配置与环境
- **风险**：本地可跑不代表可部署；外部依赖多（Postgres/Qdrant/Redis/Ollama/Embedding）。
- **升级方向**：
  - 给出 docker-compose 一键本地全栈（可选）
  - 环境变量分层：dev/staging/prod 模板

## 4. 推荐的升级路线（不改代码版）

### 4.1 未来 1-2 周（仍保持“试验期节奏”）
- 固化“默认落点 + 核心链路 CTA”
- 指标口径文档化（漏斗/看板）
- 关键异步任务：补齐超时、重试、幂等策略说明（哪怕暂时不引队列）
- 建立最小 CI（两条命令）

### 4.2 未来 1-2 月（进入“可持续迭代”）
- 任务系统上队列（Celery/RQ/Arq）
- 可观测性：日志结构化 + 指标 + tracing（至少 request_id）
- 数据权限/多租户语义确认（owner_id、admin 权限）
- 对话/AI 能力沉淀为“可配置工具与模板”（而不是纯 prompt）
